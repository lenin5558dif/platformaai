# Minimal Production Scope

## Цель

Свести текущий `PlatformaAI` к понятному и быстро запускаемому минимальному продукту для первых платящих пользователей.

Не full enterprise platform, а компактный AI SaaS с Telegram-входом, тарифами, биллингом и рабочей админкой.

## Что должно быть в минимальном проде

### 1. Единый вход и регистрация через Telegram

Нужен один сценарий:

- пользователь нажимает одну кнопку;
- открывается Telegram-бот;
- если пользователь новый, аккаунт создаётся автоматически;
- если пользователь уже существует, происходит вход в существующий аккаунт;
- после подтверждения пользователь возвращается на сайт уже авторизованным.

### 2. Простые тарифы

Нужны 4 фиксированных тарифа:

- `Free`
- `500 ₽`
- `1500 ₽`
- `5000 ₽`

Логика тарифов:

- `Free` — только бесплатные модели OpenRouter;
- платные тарифы — доступ к платным моделям;
- у пользователя должен быть понятный текущий тариф в интерфейсе;
- смена тарифа и пополнение должны быть простыми и прозрачными.

### 3. Биллинг по токенам

На платных тарифах нужна простая модель монетизации:

- берём себестоимость модели из OpenRouter;
- применяем наценку `x2`;
- списываем стоимость по фактическим токенам;
- ведём баланс пользователя;
- показываем пользователю остаток средств и расход.

Принцип расчёта:

- если OpenRouter даёт стоимость `X`, клиентская цена должна быть `2X`;
- стоимость должна считаться отдельно по input/output токенам;
- списание должно происходить после фактического ответа модели;
- перед запросом должна быть проверка, хватает ли баланса.

### 4. Каталог моделей

Для минимального релиза нужно простое поведение:

- на бесплатном тарифе показываются только free-модели OpenRouter;
- на платных тарифах показываются все разрешённые модели;
- если модель платная, а у пользователя нет оплаченного тарифа или баланса, доступ блокируется.

### 5. Админ-панель

У администратора должен быть практичный экран управления:

- количество пользователей;
- количество активных пользователей;
- общая выручка;
- общий расход токенов;
- расход за 24 часа / 7 дней;
- самые используемые модели;
- список клиентов;
- баланс клиента;
- текущий тариф клиента;
- статус клиента: активен / заблокирован.

Админка в минимальном проде не должна быть перегружена enterprise-настройками.

### 6. Понимание состояния продукта

В интерфейсе и админке должно быть понятно:

- какой у пользователя тариф;
- какой у него баланс;
- какие модели ему доступны;
- сколько он уже потратил;
- сколько пользователей в системе;
- сколько денег заработано;
- какая активность по продукту.

- есть учёт `balance`;

- есть фильтрация бесплатных моделей OpenRouter;
- есть чат и вызовы моделей через OpenRouter;
- есть базовая админ-панель со статистикой;

- есть Stripe и пополнение баланса;

- SCIM;
- SSO / OIDC;
- DLP policies;

- расширенный audit / governance контур;

Это не нужно удалять из кода прямо сейчас, но это нужно убрать из фокуса продукта и интерфейса первого релиза.

## Что сейчас не совпадает с целевым минимальным продом

### 1. Telegram-кнопка пока не делает автосоздание аккаунта

Сейчас логика рассчитана на вход в уже существующий и привязанный аккаунт.

Минимальному проду нужно другое поведение:

- Telegram должен быть первичным способом входа;
- если `telegramId` ещё не существует, пользователь должен регистрироваться автоматически;
- не должно быть требования “сначала войдите по email и потом привяжите Telegram”.

### 2. Тарифы сейчас не оформлены как жёсткая продуктовая модель

Сейчас в коде тариф чаще выглядит как `planName` в настройках пользователя, а не как строгая тарифная сущность.

Для минимального прода нужно:

- фиксированный список тарифов;
- понятная привязка тарифа к доступу к моделям;
- явная логика лимитов и баланса;
- понятное отображение в UI и админке.

### 3. Наценка сейчас не зафиксирована как `x2`

Сейчас расчёт идёт через переменную `OPENROUTER_MARKUP`, а дефолт равен `1.0`.

Для минимального прода нужно:

- установить продуктовый дефолт `x2`;
- явно использовать его в расчёте стоимости;
- согласовать отображение цены и внутреннего списания.

### 4. Бесплатный доступ сейчас привязан к нулевому балансу, а не к тарифу

Сейчас логика ближе к:

- есть баланс -> можно использовать платные модели;
- нет баланса -> только free models.

Для минимального прода это надо сделать понятнее:

- `Free` — отдельный тариф;
- платные тарифы — отдельные тарифы;
- доступ определяется не только балансом, но и тарифом.

### 5. Админка шире, чем нужно для первого релиза

Сейчас админка уже богаче MVP. Это хорошо для будущего, но для минимального прода стоит сфокусироваться на 2 разделах:

- `Dashboard`
- `Clients / Billing`

Остальные enterprise-разделы можно временно скрыть из основного навигационного сценария.

## Рекомендуемая версия минимального продукта

### Пользовательский сценарий

1. Пользователь заходит на сайт.
2. Нажимает одну кнопку `Войти через Telegram`.
3. Бот подтверждает вход.
4. Если аккаунта нет, он создаётся автоматически.
5. Пользователь попадает в чат.
6. Видит свой тариф, баланс и доступные модели.
7. На `Free` использует только бесплатные модели.
8. При оплате получает баланс и доступ к платным моделям.

### Сценарий администратора

1. Администратор открывает `/admin`.
2. Видит:
   - пользователей;
   - активность;
   - токены;
   - выручку;
   - топ моделей.
3. Открывает `/admin/clients`.
4. Видит:
   - клиента;
   - баланс;
   - тариф;
   - активность;
   - статус.
5. Может:
   - поменять тариф;
   - заблокировать клиента;
   - скорректировать баланс при необходимости.

## На чём мы остановились сейчас

По текущему состоянию кода:

- база для минимального прода уже есть;
- ключевой недостающий кусок — Telegram-first auth с автосозданием аккаунта;
- второй важный кусок — жёсткая продуктовая модель тарифов;
- третий — упрощение интерфейса и навигации под простой SaaS, а не под enterprise-комбайн.

## Рекомендуемый порядок реализации

### Этап 1. Вход

- переделать Telegram flow в `login-or-signup`;
- убрать зависимость от email-first сценария;
- сделать Telegram основным способом входа.

Статус: завершен.

Этап 1 в рекомендуемом порядке реализации закрыт: вход через Telegram уже работает как основной сценарий.

Что реализовано:

- Telegram login flow создает пользователя автоматически при первом входе;
- вход через Telegram работает как отдельный продуктовый сценарий без обязательной предварительной привязки;
- локальный стенд и бот подняты для ручной проверки первого этапа.

### Этап 2. Тарифы

- ввести 4 фиксированных тарифа;
- зафиксировать доступные модели по тарифу;
- зафиксировать продуктовую наценку `x2`.

Статус: завершен.

Что реализовано:

- в коде зафиксированы 4 тарифа: `Free`, `500 ₽`, `1500 ₽`, `5000 ₽`;
- бесплатный тариф показывает только free-модели OpenRouter;
- платные тарифы открывают доступ к платным моделям;
- в интерфейсе и админке отображается текущий тариф пользователя.

### Этап 3. Биллинг

- привести расчёт токенов и списаний к единой схеме;
- показать баланс и расход в UI;
- привязать оплату к тарифам и пополнениям.

Статус: завершен.

Что реализовано:

- продуктовый дефолт для наценки на OpenRouter зафиксирован как `x2`;
- расчёт стоимости и резервирование квоты приведены к одной схеме;
- в UI и API добавлены понятные подписи для баланса, расходов и транзакций;
- пополнение через Stripe привязано к фиксированным тарифным пакетам.

### Этап 4. Интерфейс

- упростить пользовательский путь до:
  `вход -> чат -> тариф -> баланс -> оплата`;
- скрыть лишние enterprise-разделы;
- оставить компактную админку.

### Этап 5. Админка

- оставить ключевую сводку;
- оставить клиентов и финансы;
- проверить, что админ видит выручку и usage по моделям.

## Краткий вывод

Минимальный прод `PlatformaAI` — это:

- вход через Telegram в один клик,
- чат с OpenRouter,
- 4 понятных тарифа,
- free-only режим на бесплатном тарифе,
- списание по токенам с наценкой `x2`,
- баланс и платежи,
- короткая и полезная админка.

Всё остальное можно считать вторым этапом, а не блокером первого релиза.
