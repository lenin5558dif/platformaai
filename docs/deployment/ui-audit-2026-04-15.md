# UI Audit Report: 2026-04-15

Scope:
- Live deploy server from `server.md`
- Public URL: `http://72.56.90.36:1717`
- Audit method: Playwright visual pass + SSH verification on the deploy server

## Summary

The production UI is visually coherent on public screens, but access behavior is inconsistent between routes. The most serious issue is the login flow: the live `/login` screen currently shows that sign-in methods are unavailable, which blocks normal access to the private product.

## Confirmed Environment

From the deploy server:

- `NEXTAUTH_URL=http://72.56.90.36:1717`
- `NEXT_PUBLIC_APP_URL=http://72.56.90.36:1717`
- `APP_URL=http://72.56.90.36:1717`
- `NODE_ENV=production`
- `AUTH_BYPASS=0`
- `NEXT_PUBLIC_TELEGRAM_AUTH_ENABLED=1`

Container status was healthy during the audit and the app container was up.

## Main Findings

### 1. Login flow is effectively blocked

Observed on `http://72.56.90.36:1717/login?mode=signin`:

- UI renders the message `Сейчас недоступны способы входа`
- Telegram also falls back to unavailable state
- Normal email/password form was not visible in the live deploy

Impact:
- Users cannot rely on the web login screen to access protected pages

Notes:
- This does not match the current repository code in `src/lib/auth-ui.ts` and `src/components/auth/LoginForm.tsx`, so the deployed runtime behavior should be rechecked against build-time config and auth capability resolution.

### 2. Guest homepage renders a full chat shell but backend calls fail with 401

Observed on `/`:

- The app shows a ready-looking chat interface
- Browser console shows `401 Unauthorized` for:
  - `/api/chats`
  - `/api/models`
  - `/api/me`

Impact:
- Users see a product state that looks available, but core data is unauthorized
- This is misleading and should either degrade more clearly or redirect to login

### 3. Access behavior is inconsistent across protected routes

Observed without a session:

- `/models` and `/pricing` render full `AppShell` UI
- `/billing`, `/profile`, `/settings`, `/timeline`, `/events` render `... недоступно`
- `/audit` and `/admin` render `Доступ запрещен`

Impact:
- Protected surfaces do not present a consistent access policy
- Some pages visually imply an authenticated user even when no session exists

### 4. Mobile sidebar overlay on chat needs refinement

Observed on `/` at mobile width:

- Opening the sidebar leaves the composer visible underneath
- Overlay layering and interaction boundaries feel unfinished

Impact:
- The off-canvas navigation looks fragile and can confuse taps/focus

### 5. Invite accept invalid-token state is not explicit enough

Observed on `/invite/accept?token=test-invalid`:

- The screen presents a normal acceptance flow with CTAs
- A clearly invalid-token state was not surfaced during the public audit

Impact:
- Users may think the invite is still actionable when the token is bad or expired
